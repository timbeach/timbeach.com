# timbeach.com — End-to-End System Architecture

> **Private doc.** Lives in `docs/`, which `.deployignore` keeps off the public
> web root. Versioned in git, never published. This is the single map of how the
> whole site works: authoring → rendering → audio → social → feed → deploy →
> scheduled release. For the deeper TTS treatment see [`tts-pipeline.md`](./tts-pipeline.md).

---

## 1. What this is

A static personal site at **timbeach.com**, served as plain files from
`vultr:/var/www/timbeach.com`. No server-side application, no database, no build
step in the traditional sense. The browser loads a thin HTML shell and a small
amount of vanilla ES-module JavaScript that renders everything client-side from
a JSON registry of articles.

Two design commitments shape everything below:

1. **Vanilla, no dependencies, no bundler.** Files on disk are the files in the
   browser. You can open `index.html` directly or serve the folder with any
   static server.
2. **Pre-render anything expensive at publish time, not in the reader's
   browser.** Audio (TTS), social preview images, and the RSS feed are all
   generated by author-side Python tools and shipped as static artifacts. Every
   reader gets identical, instant results regardless of their OS or hardware.

## 3. Content model

An article is **two things**: a markdown body in `articles/<slug>.md` and a
registry entry in `articles/articles.json` keyed by the `.md` filename.

```json
"the-slug.md": {
  "title": "Human Title",
  "date": "2026-06-07",          // ISO; drives BOTH date gates (§8)
  "tags": ["a", "b"],
  "emoji": "📋",
  "summary": "One substantive sentence — homepage card + RSS <description>.",
  "section": "override-label",   // optional; else derived from first tag
  "hero": "pix/card.png",         // optional; OG/share-card source (§7)
  "audio": "audio/the-slug.ogg",          // written by render tool
  "timings": "audio/the-slug.timings.json", // written by render tool
  "voice": "bm_lewis",                     // written by render tool
  "duration": 303.659                      // written by render tool
}
```

**No frontmatter** lives in the markdown — all metadata is in the JSON. The
`audio`/`timings`/`voice`/`duration` fields are populated automatically by
`render_article.py`; an article without them simply shows no read-aloud button
(graceful degradation).

## 4. Client rendering (the SPA)

`index.html` is a static shell. A `<script type="module">` boots `app.js`, which
registers route handlers and calls `initRouter()`.

**Routing** (`router.js`) is hash-based, so the server only ever serves
`index.html`:

| Hash | Route |
|---|---|
| `#/` (or `#`, empty) | Home |
| `#/article/<slug>` | Article reading view |
| `#/music` | Music page |
| `#/about` | About page (mounts the starfield) |
| `#articles/<slug>.md` | **Legacy** → silently `replaceState`'d to `#/article/<slug>` |

**Homepage** (`app.js → loadArticles`) fetches `articles.json`, maps it to an
array, **filters by the homepage date gate** (§8), sorts by date descending, and
renders cards. Each card shows the emoji, title, derived section, short date,
and `summary`.

**Article view** (`article.js`) fetches `articles.json` for metadata and the
`.md` body, runs `parseMarkdown`, and renders the reading view (plus the
read-aloud bar if audio metadata exists, a copy-share-link button, and an image
lightbox).

**Other modules:** `theme.js` (day/night with `localStorage` + OS fallback,
applied by an inline script in `<head>` to avoid flash), `starfield.js`
(astronomical starfield + sky-info widget, init/destroy on the `/about` route
only), `tts.js` (read-aloud transport — see §6).

## 5. The markdown parser parity contract ⚠️

There are **two** markdown parsers in this system:

- `parseMarkdown` in `js/article.js` — renders articles in the browser.
- `extract_paragraphs` in `tools/render_article.py` — segments text for TTS.

The TTS timings sidecar maps audio time ranges to **paragraph indices**. For
paragraph highlighting to stay in sync, both parsers MUST agree on paragraph
boundaries. The deploy gate (`render_article.py --validate`) asserts this parity
and **aborts the deploy on divergence**.

> **Do not refactor `parseMarkdown` without re-rendering all audio.** Known
> divergences already fixed: `# → <h2>` title mapping and per-line `<p>`
> wrapping. If a third appears, the long-term fix is to extract the parser into
> shared JS and validate via Node. See `tts-pipeline.md` for the full story.

## 6. TTS / read-aloud pipeline

Audio is **pre-rendered at publish time** on the author's machine, never
synthesized in the reader's browser (Web Speech is unreliable on Linux/Brave;
in-browser Kokoro WASM has poor real-time-factor on mid-range laptops).

- `tools/render_article.py <slug> --voice <id>` synthesizes with **Kokoro**
  (ONNX), producing `audio/<slug>.ogg` (Opus) + `audio/<slug>.timings.json`, and
  writes `audio`/`timings`/`voice`/`duration` back into `articles.json`.
- Default voice `bm_lewis`. Prefix `b*` → en-gb, `a*` → en-us (auto-inferred).
- The client streams the Opus via a native `<audio>` element; `timeupdate`
  events drive paragraph highlighting from the timings sidecar.
- `--validate` checks timings against current markdown (the deploy gate).

One-time setup, failure modes, and voice list: see
[`tts-pipeline.md`](./tts-pipeline.md) and `CLAUDE.md`.

## 7. Social share pages

Social crawlers don't run JS and never see the hash fragment, so a shared
`#/article/<slug>` URL would always resolve to `index.html`'s generic OG image.
To fix this, `build_share_pages.py` generates a **crawlable page per article**:

- Writes `a/<slug>/index.html` with that article's `og:`/`twitter:` tags plus a
  JS + `<meta http-equiv="refresh">` redirect into the SPA at `#/article/<slug>`.
- **Share the `https://timbeach.com/a/<slug>/` URL**, not the `#` URL. Humans get
  redirected to the reader view; crawlers read the baked-in tags.
- **og:image** is always a crisp **1200×630** `a/<slug>/og-<hash>.png` (rendered
  with Pillow). Source precedence: explicit `hero` → first body image → branded
  text card. Local images are contain-fit onto the dark canvas via Lanczos;
  a remote `hero` URL is passed through verbatim.
- Filenames are content-hashed, so a changed image orphans the old hash. deploy
  does a `--delete` pass scoped to `a/` to keep it an exact mirror (§ deploy).
- After deploying a new/edited article, re-scrape in the LinkedIn Post Inspector
  to bust cached previews.

`a/` is gitignored — it's a build artifact, rebuilt every deploy.

## 8. Date-based gating — TWO independent gates

A single field, `date` (ISO `YYYY-MM-DD`), drives two **completely separate**
gates. This is the heart of how scheduling works, and the part most easily
misunderstood.

### Gate A — Homepage (client-side, no deploy needed)

`app.js → loadArticles` filters `a.date <= todayLocal`, where `todayLocal` is
computed from the **viewer's own clock**. A future-dated article is already on
the server but stays off the homepage list until the viewer's calendar reaches
its date — then it appears with no deploy required. String compare is
chronological because both sides are ISO.

### Gate B — RSS feed (deploy-time)

`build_feed.py` runs during `deploy.sh` and emits whatever articles exist at
that moment (it does not itself filter by date — it reflects the repo state at
deploy time). So a scheduled post **only enters `feed.xml` on a deploy that
happens on or after its date.** Without a deploy, subscribers never see it.

### The consequence

- The homepage "reveals" a scheduled post automatically on its date.
- The RSS feed does NOT — it needs a deploy on/after that date.
- That gap is exactly what the auto-deploy system (§9) closes.

> A future `date` therefore delays both surfaces, but by different mechanisms:
> the homepage by client clock, the feed by "has a deploy happened since the
> date." The direct article URL (`#/article/<slug>`) and the share page
> (`a/<slug>/`) are NOT date-gated — once deployed, they're reachable by link.

## 9. Deployment & scheduled release

### `deploy.sh` (manual or invoked by the auto-deployer)

CWD **must** be the repo root (`rsync` uses `$PWD/` as source). Steps:

1. **Validate** — `render_article.py --validate` over all articles. Stale
   timings abort the deploy (catches "edited an article, forgot to re-render").
2. **Build feed** — `build_feed.py` → `feed.xml`.
3. **Build share pages** — `build_share_pages.py` → `a/<slug>/`.
4. **rsync** the repo to `vultr:/var/www/timbeach.com`, excluding everything in
   `.deployignore`.
5. **Prune `a/`** — a second `rsync --delete` scoped to `a/` removes orphaned
   content-hashed share-card images.

`.deployignore` is the single source of truth for what must never go public:
internal `*.md` at root, `STATUS-*.md`, `*.sh`, `/docs/`, `/tests/`,
`counter-data/`, `.beads/`, `.claude/`, `tools/`, `archive/`, `.git/`, etc. The
whole repo is rsynced, so **anything not excluded here goes public** — keep it
current when adding internal files.

### Scheduled release: cron → anacron → `timbeach-autodeploy`

Gate B means scheduled posts need a deploy on/after their date to reach the
feed. This chain automates that:

```
crontab  ──fires──▶  anacron  ──runs once/day──▶  timbeach-autodeploy  ──▶  deploy.sh
(@reboot + 05:00)    (catch-up after off)         (guarded: only if needed)
```

- **cron** (`crontab -l`) only *triggers* anacron: once `@reboot` and once at
  `0 5 * * *`. cron alone can't catch up missed runs on a laptop that was off.
- **anacron** (`~/.config/anacrontab`, job `timbeach-autodeploy`, period 1 day)
  keeps a per-job timestamp in `~/.local/var/spool/anacron/` and runs the job at
  most once per day, **catching up after the machine was powered off** (travel,
  sleep) — the property plain cron lacks. No systemd timers (Aegix uses runit;
  cron is cronie, runit-supervised).
- **`timbeach-autodeploy`** (`~/.local/bin/`) is the guard that makes the daily
  run cheap and safe:
  - Re-execs itself under `flock -n` so overlapping ticks can't collide.
  - Computes the **due** set: articles with `date <= today` and markdown present.
  - Fetches the **live** feed (`curl` the deployed `feed.xml`) and extracts the
    set of article slugs currently in it (from `<link>` tags only — not
    `content:encoded`, whose embedded inter-article links would pollute the set).
  - If `due == live`: exit silently (the common case — costs ~one curl).
  - If they differ: run `./deploy.sh`, logging to
    `~/.local/var/log/timbeach-autodeploy.log`.
  - If the feed is unreachable (offline): skip cleanly — a deploy would fail at
    rsync anyway.

> **Side effect to know:** the auto-deployer publishes ANY due-dated article in
> the local repo. Keep not-ready drafts **future-dated** (or dateless) so they
> don't auto-ship on the next 05:00 run.

So to schedule a post: set its `date` to a future day and commit/leave it in the
repo. The homepage reveals it on that day client-side; the next auto-deploy
on/after that day pushes it into the RSS feed. No manual action required.

## 10. Authoring workflow (the `publish-article` skill)

Day-to-day publishing goes through the `publish-article` Claude Code skill
(`~/.claude/skills/publish-article/SKILL.md`, run `/publish-article <path>`).
Its purpose framing: **these articles demonstrate expertise and competence in
public** — internal editorial guidance only, never stated in the prose. Ordered
steps:

1. Acquire content (file path or inline; require ≥1 `#` heading).
2. Handle **body** images → copy to `pix/`, rewrite paths to `pix/<file>`.
3. Propose metadata (title, date, tags, emoji, **summary**) for approval.
4. Write `articles/<slug>.md` + extend `articles.json`. **No audio yet.**
5. **Local preview & lock** — serve, review at `#/article/<slug>`, iterate
   cheaply (no timings to invalidate), lock the content.
6. **Hero image** (optional, post-lock) → `pix/` + `hero` field.
7. **Render audio** against locked content (`render_article.py <slug> --voice`).
8. **Validate parity** (`--validate <slug>`).
9. **Set go-live date** — today, or a future `YYYY-MM-DD` (drives both gates §8).
10. `./deploy.sh`.
11. Git commit + push the specific changed files (no Co-Authored-By trailers —
    a repo hook blocks them).

The deliberate ordering — preview/lock **before** the ~3-min audio render —
exists so content edits don't waste renders. Hero is post-lock because it's a
share-card concern, not body content.

## 11. Operational quick reference

| Task | Command |
|---|---|
| Local preview | `python3 -m http.server 8000` → `http://localhost:8000/#/article/<slug>` |
| Render audio | `tools/venv/bin/python tools/render_article.py <slug> --voice bm_lewis` |
| Re-render (force) | `… render_article.py <slug> --voice <id> --force` |
| Validate all timings | `tools/venv/bin/python tools/render_article.py --validate` |
| Build feed only | `tools/venv/bin/python tools/build_feed.py` |
| Build share pages only | `tools/venv/bin/python tools/build_share_pages.py` |
| Full deploy | `./deploy.sh` (from repo root) |
| Watch auto-deploy | `tail -f ~/.local/var/log/timbeach-autodeploy.log` |
| Dry-run the release guard | run `~/.local/bin/timbeach-autodeploy` manually; it's idempotent |
| Inspect anacron schedule | `cat ~/.config/anacrontab` · `crontab -l` |

### Common gotchas

- **Deploy aborts on validate** → an article's markdown changed after its audio
  was rendered. Re-render with `--force`, then validate.
- **Read-aloud bar hides itself** (`paragraph count mismatch` / `first paragraph
  text mismatch` in console) → same cause; re-render with `--force`.
- **Validate passes but bar still hides** → client/server parser divergence
  (§5). See `tts-pipeline.md`.
- **A draft auto-shipped** → it was due-dated (`date <= today`) and present in
  the repo; the 05:00 auto-deploy released it. Keep drafts future-dated.
- **Shared link shows generic image** → you shared the `#/article/<slug>` URL
  instead of `a/<slug>/`; and/or the crawler cached an old preview (re-scrape).
- **Something internal leaked to the web** → it wasn't in `.deployignore`. Add
  it; the whole repo is rsynced by default.

## 12. Server topology

- **Host:** `vultr` (SSH alias). **Web root:** `/var/www/timbeach.com`.
- Plain static file serving. The only "dynamic" server-side state is the
  site-counter data under `counter-data/` (created server-side, never committed;
  see `docs/superpowers/plans/2026-06-07-site-counter.md`).
- All scheduling/release intelligence lives **client-side** (homepage gate) and
  **on the author's laptop** (cron/anacron auto-deploy) — the server itself is
  dumb by design.
